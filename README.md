# B2B SaaS Revenue Systems Handbook

顧客管理・請求・会計のデータが「どうしてもつながらない」という経験は、B2B SaaS に関わるエンジニアや RevOps 担当者なら誰でも覚えがあるはずです。ツールを変えても、担当者を増やしても、問題は解決しない。それはツールや組織の問題ではなく、構造的な必然だからです。

本書は、この「なぜつながらないのか」を出発点に、B2B SaaS の収益系システムを正しく設計するための体系を提供します。

中心に置いたのは、業務オブジェクトを4つの型——交渉・コミットメント・履行・決済——に分類する枠組みです。この分類を理解すると、見積・契約・請求書・入金などの性質——可変か不変か、どのシステムが正本を持つべきか——を自分で導出できるようになります。各ドメインの責務も、ビジネスモデルや成長フェーズに応じたシステム構成パターンも、すべてこの型分類から導かれます。

対象は B2B SaaS（SLGとPLGの両方）です。HubSpot・Salesforce・Stripe などは文脈を共有するための例として登場しますが、特定ツールの使い方を説明するものではありません。業務の構造と、その設計判断の根拠を理解すること——それが本書の目的です。

---

## 対象読者

本書が想定する読者は、B2B SaaS の収益系システムに何らかの形で関わる実務家です。データエンジニアや Analytics Engineer として請求データを扱っている方、RevOps・CRM・ERP の担当者として複数システムの連携に悩んでいる方、業務システムエンジニアとして設計の根拠を言語化したいと感じている方——職種は異なっても、「なぜこの構造になっているのか」を理解したい、という問いを持つ人に向けて書きました。

---

## 読み方

第0章から読み始めることを勧めます。「なぜ顧客管理・請求・会計のデータモデルは統合できないのか」——この問いへの答えを理解すると、残りの章がすべてその帰結として読めるようになります。

第1章（原理）では、業務オブジェクトを4つの型に分類する枠組みと、本書を通じて使う基礎概念を整理します。この分類が、以降の章すべての議論の土台になります。

第2章（ドメイン）では、顧客・商品・見積・契約・請求書・入金といった各ドメインを「なぜ存在するのか」から説明します。第1章の型分類の文脈で読むと、各ドメインの設計判断の理由が見えてきます。

第3章（業務プロセス）では、これらのドメインが実際の業務フロー——Lead-to-Cash、Signup-to-Expansion、更新と解約——の中でどう連鎖するかを説明します。ドメイン単体ではなく、流れとして理解することが目的です。

第4章（システム構成パターン）では、特定のツール群ではなく、ビジネスモデルと成長フェーズに対応したシステム構成のパターンを説明します。スタートアップ初期からエンタープライズ、PLGまでをカバーします。

第5章（データ分析）では、業務システムのデータを分析基盤に取り込む際のデータモデル設計を扱います。収益分析に必要なファクトテーブルとディメンションテーブルの設計を、業務の文脈とともに説明します。
