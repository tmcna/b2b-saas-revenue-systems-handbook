# B2B SaaS Revenue Systems Handbook

「今月の売上を教えてください」——この問いに、営業・経理・データアナリストがそれぞれ答えると、三者三様の数字が返ってくることがあります。誰も嘘をついていないのに、なぜか数字が合いません。

数字が合わない背景には、B2B SaaS の収益系システムに共通する構造的な理由があります。本書はその構造を理解するための枠組みを提供します。業務ドメインの分類から、各システムの責務、ビジネスモデルに応じた構成パターンまで、設計判断の根拠を自分で導き出せるようになることを目指しています。

---

## 対象読者

本書が想定する読者は、B2B SaaS の収益系システムに何らかの形で関わる実務家です。データエンジニアや Analytics Engineer として請求データを扱っている方、RevOps・CRM・ERP の担当者として複数システムの連携に悩んでいる方、業務システムエンジニアとして設計の根拠を言語化したいと感じている方——職種は異なっても、「なぜこの構造になっているのか」を理解したい、という問いを持つ人に向けて書きました。

---

## 読み方

序章から読み始めることを勧めます。「なぜ顧客管理・請求・会計のデータモデルは統合できないのか」——この問いへの答えを理解すると、残りの章がすべてその帰結として読めるようになります。

第1章（業務システムの構成要素）では、業務ドメインをフローエンティティ・マスターデータ・計算サービス・台帳の4種類に分類する枠組みと、本書を通じて使う基礎概念を整理します。この分類が、以降の章すべての議論の土台になります。

第2章（各ドメインの責務と設計）では、顧客・商品・見積・契約・請求書・入金といった各ドメインを「なぜ存在するのか」から説明します。第1章の分類の文脈で読むと、各ドメインの設計判断の理由が見えてきます。

第3章（業務フローの全体像）では、これらのドメインが実際の業務フロー——Lead-to-Cash、Signup-to-Expansion、更新と解約——の中でどう連鎖するかを説明します。ドメイン単体ではなく、流れとして理解することが目的です。

第4章（ビジネスモデル別の構成パターン）では、特定のツール群ではなく、ビジネスモデルと成長フェーズに対応したシステム構成のパターンを説明します。スタートアップ初期からエンタープライズ、PLGまでをカバーします。

第5章（収益分析のデータモデル設計）では、業務システムのデータを分析基盤に取り込む際のデータモデル設計を扱います。収益分析に必要なファクトテーブルとディメンションテーブルの設計を、業務の文脈とともに説明します。

結語（あとがきにかえて）では、本書を書くに至った経緯と、著者自身が考える「収益系システムの分断とどう向き合うか」を記しています。

付録では、本文を読みながら参照できる対応表・用語集・プロダクト一覧を収録しています。

## 法令・会計基準の参照時点

本書の法令・会計基準に関する記述は、**2026年6月時点**の内容に基づいています。主な参照法令・基準は次のとおりです。

| 法令・基準 | 主な参照箇所 |
|---|---|
| 会社法（第432条・第439条） | 会計帳簿の作成・保存義務、計算書類の確定 |
| 法人税法施行規則（第59条） | 帳簿書類の保存期間（原則7年） |
| 消費税法（第57条の4）・同施行令（第50条） | 適格請求書等保存方式（令和5年10月施行）、帳簿の保存方法 |
| 電子帳簿保存法 | 電子取引データの訂正・削除要件（令和4年1月施行の改正を含む） |
| 金融商品取引法 | 内部統制報告制度（会計期間確定後の修正禁止） |
| 企業会計基準第29号「収益認識に関する会計基準」 | 収益認識（2021年4月1日以後開始事業年度から強制適用） |
| IFRS 15「顧客との契約から生じる収益」 | 国際会計基準に基づく収益認識 |
| 民事訴訟法 | 契約記録の証拠保全（一般原則） |

法令は改正されることがあります。本書の記載が最新の法令に対応しているかどうかは随時ご確認ください。
