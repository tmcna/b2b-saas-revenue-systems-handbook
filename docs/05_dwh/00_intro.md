# 収益分析のデータモデル設計——何をDWHに乗せるべきか

## なぜオペレーショナルシステムだけでは足りないのか

[第0章](../00_introduction/02_why_systems_fragment.md)で説明したように、B2B SaaSの収益系システムは構造的に3つのドメインに分断されています。CRMは営業ドメインの、Billingは請求ドメインの、ERPは財務ドメインの正本をそれぞれ持ち、互いに独立して動きます。

この分断は設計上の正しい選択です。しかし分析の観点からは問題になります。「この顧客セグメントの解約率と請求金額の関係は？」「プランごとのNRRはどう違うか？」——こうした問いに答えるには、複数のシステムに散らばったデータを横断する必要があります。どのオペレーショナルシステムも、単独ではこれらの問いに答えられません。

データウェアハウス（DWH）は、この問題を解決するために存在します。

---

## DWHに乗せるべきデータの判断基準

「どのデータをDWHに取り込むべきか」は、「そのシステムに存在するかどうか」では決まりません。以下の3つの条件のいずれかを満たすとき、DWHへの取り込みが必要になります。

### 条件1：複数システムをまたぐ問いに答えるため

CRM・Billing・ERPのいずれか1つでは答えられず、複数のシステムのデータを結合しなければ答えられない問いがあります。

```
例：「商談クローズから初回入金までの期間は、顧客セグメント別にどう異なるか？」

  CRM        → 商談クローズ日・顧客セグメント
  Billing    → 初回請求日
  ERP        → 初回入金日

  → 3システムのデータを結合しないと答えられない
```

オペレーショナルシステムはドメインの境界で分断されています。DWHはその境界を越えて問いに答えるための場所です。

### 条件2：時系列の問いに答えるため

オペレーショナルシステムは「今の状態」を管理するために設計されています。契約が更新されれば古い内容は上書きされ、顧客の属性が変わればそれ以前の状態は失われます。

「3ヶ月前の時点でのMRRはいくらだったか」「昨年獲得した顧客コホートの継続率はどう推移しているか」——こうした問いに答えるには、過去の状態のスナップショットが必要です。DWHはこの履歴を保持します。


| オペレーショナルシステム | DWH |
|---|---|
| 現在の状態のみ保持|月次スナップショットとして過去の状態を蓄積 |
| 契約更新 → 旧内容を上書き |更新前後の状態を両方保持 |

### 条件3：派生指標の計算のため

MRR・NRR・コホート継続率などの指標は、単一の業務イベントに対応するレコードとして存在しません。複数のイベントにまたがる計算によって初めて導出される派生指標です。

オペレーショナルシステムにこれらの計算を組み込むと、本来の業務処理とは無関係な集計ロジックが混入し、システムの責務が曖昧になります。派生指標の計算はDWH層の責務です。

---

## 逆に、DWHに乗せる必要がないデータ

上記3条件のいずれも満たさないデータは、DWHに取り込む優先度が低くなります。

**オペレーショナルシステム内で完結する現在状態**　——　「この顧客のサブスクリプションは現在アクティブか」という問いは、BillingシステムがリアルタイムのAPIで答えるべきです。DWHは更新ラグがあるため、業務上のリアルタイム判断には使えません。

**ワークフロー中間状態**　——　「見積のドラフト」「承認待ちの注文」といった一時的なプロセス状態は、ワークフローが完了すれば意味を失います。分析上の価値が低く、DWHに蓄積する必要はありません。

**単一システムで完結する問い**　——　「今月発行した請求書の合計金額」はBillingシステムが直接答えられます。わざわざDWHに取り込まなくても、オペレーショナルシステムのレポート機能で十分な場合があります。

---

## 業務システムとDWHは、同じエンティティに別のモデルを要求する

実務でよく起きる混乱として、**業務システムとDWHのデータモデルを同一視してしまう**ことがあります。「DWHはソースDBのコピーでいい」という発想です。しかし、同じエンティティでも目的が違えば、正しいモデルは変わります。

顧客（Customer）を例にとります。

**業務システム（CRM）における顧客**は、「今この顧客はどういう状態か」を管理するためにあります。顧客ティアが SMB から Enterprise に変わったら `UPDATE` で上書きするのが正しい。営業担当者は現在の正しい状態を参照できれば十分です。ここでは CRUD が適切なモデルです。

**DWH における顧客ディメンション**は、「あの請求が発行されたとき、この顧客はどういう状態だったか」に答えるためにあります。「スタートアップだった顧客の解約率」を分析するには、請求日時点での顧客ティアが必要です。CRM が `UPDATE` で上書きした後では、過去の状態は復元できません。ここでは SCD Type 2 が適切なモデルです。

```
同じ「顧客」でも：

  CRM（業務システム）：  tier = 'SMB'  → UPDATE → tier = 'Enterprise'  （上書き）
  DWH（ディメンション）：  tier = 'SMB',  valid_to = '2024-06-30'         （履歴保持）
                          tier = 'Enterprise', valid_from = '2024-07-01'  （新レコード）
```

この違いは、各システムの責務の違いから来ています。業務システムは「現在を正確に動かす」ために設計され、DWHは「過去を正確に問い合わせる」ために設計されます。同じエンティティを扱っていても、目的が異なるため、正しいデータモデルも異なります。

**DWHにおける SCD Type 2 の実装** は、CRM の CRUD とは別のしくみで変更を捕捉する必要があります。CRM が `UPDATE` した後では変更前の値は失われているためです。主なアプローチは、DB のトランザクションログを読む CDC（Change Data Capture）と、定期的な全件スナップショットの差分比較（dbt snapshot など）の2つです。詳細は[顧客ディメンションテーブル](04_dim_customer.md)で説明します。

---

## 列を追加する前に「何のための値か」を定義する

DWH のテーブルに列を追加するとき、「あると便利そう」という理由だけで足すのは避けてください。列の意味が曖昧なまま使われ始めると、見る人によって異なる解釈が生まれ、分析結果に一貫性がなくなります。

追加前に決めておくべきことは2つです。「**何の問いに答えるための値か**」と「**何と比較するための値か**」です。

`list_price`（定価）を例にとります。割引率を分析したいという動機は明確ですが、「定価」の定義は自明ではありません。標準価格表の価格なのか、この顧客に適用した価格表の価格なのかで、割引率の意味が変わります。定義を確定する前に列だけ作ると、後から「この列は何の定価？」という混乱が生じます。

これはどの列にも当てはまる原則です。列を追加するときは、その定義をデータカタログや dbt の description に明記し、用途を見れば意味がわかる状態にしてから追加してください。

## 本章が扱うテーブル

以上の基準に照らして、本章では次のテーブルを扱います。

| テーブル | 取り込む理由 |
|---|---|
| 請求明細ファクト | Billing単体では商品・顧客・期間を横断した集計が難しい |
| 入金ファクト | 請求と入金のマッチングはBillingとERPをまたぐ |
| 見積明細ファクト | 受注率・割引傾向の分析はCRMとBillingをまたぐ |
| 顧客ディメンション | 複数システムに散在する顧客属性を統合する |
| 商品ディメンション | 商品の分類・属性を分析軸として整備する |
| 売掛金エイジング | 未収金の時系列管理はERPの現在状態では追えない |
| MRRスナップショット | サブスクリプション状態の時系列スナップショット |
| コホートファクト | 顧客獲得時期を起点にした時系列の継続率計算 |
| NRRファクト | 複数月のMRRスナップショットから導出される派生指標 |
| 収益認識スケジュールファクト | 請求ベースでなくサービス提供期間に基づく認識売上・繰延収益残高を管理する |

契約や注文はオペレーショナルシステム（CRM・Billing）の中で管理される業務エンティティです。ただし「契約期間別のチャーン傾向」や「注文から請求までのリードタイム分析」が必要になった場合は、これらも取り込みの対象になります。本章では収益の流れを直接測定するテーブルに絞って説明します。

---

## パターン別の適用可否

各テーブルがどの構成パターン（[第4章](../04_patterns/)）で使われるかを整理します。◎は特に重要、○は使用可、△は使えるが優先度低、×は対象外または意味が薄い段階です。

| テーブル | Early SLG | Startup SLG | Enterprise SLG | PLG |
|---|:---:|:---:|:---:|:---:|
| 請求明細ファクト | △ | ○ | ○ | ○ |
| 入金ファクト | △ | ○ | ○ | ○ |
| 見積明細ファクト | × | ○ | ◎ | × |
| 顧客ディメンション | △ | ○ | ○ | ○ |
| 商品ディメンション | △ | ○ | ○ | ○ |
| 売掛金エイジング | × | △ | ◎ | × |
| MRRスナップショット | △ | ○ | ○ | ○ |
| コホートファクト | × | ○ | ○ | ◎ |
| NRRファクト | × | ○ | ◎ | ○ |
| 収益認識スケジュールファクト | × | △ | ◎ | △ |

**Early Stage SLG** では、見積・契約台帳がスプレッドシートで管理されており、構造化データとしてDWHに取り込むことが困難なテーブルがあります。顧客数も少ないため、コホート分析やNRRは統計的な意味が薄くなります。DWHの本格稼働はStartup SLG以降が現実的です。

**見積明細ファクト**はSLGにのみ適用されます。PLGではセルフサービスで契約が完結するため、見積プロセスが存在しません。

**売掛金エイジング**はEnterprise SLGで特に重要です。年額前払いや請求書払いが主流であり、延滞管理・貸倒引当金の計算が業務として成立しています。Startup SLGではカード自動課金の割合が高く、優先度は下がります。

**コホートファクト**はPLGで最も重要です。セルフサービスによる自然なチャーンを把握するうえで、コホート別の継続率が主要な指標になります。
