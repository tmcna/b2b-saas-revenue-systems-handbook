# 収益分析のデータモデル設計——何をDWHに乗せるべきか

## なぜオペレーショナルシステムだけでは足りないのか

[第0章](../00_principles/02_why_systems_fragment.md)で説明したように、B2B SaaSの収益系システムは構造的に3つのドメインに分断されています。CRMは営業ドメインの、Billingは請求ドメインの、ERPは財務ドメインの正本をそれぞれ持ち、互いに独立して動きます。

この分断は設計上の正しい選択です。しかし分析の観点からは問題になります。「この顧客セグメントの解約率と請求金額の関係は？」「プランごとのNRRはどう違うか？」——こうした問いに答えるには、複数のシステムに散らばったデータを横断する必要があります。どのオペレーショナルシステムも、単独ではこれらの問いに答えられません。

データウェアハウス（DWH）は、この問題を解決するために存在します。

---

## DWHに乗せるべきデータの判断基準

「どのデータをDWHに取り込むべきか」は、「そのシステムに存在するかどうか」では決まりません。以下の3つの条件のいずれかを満たすとき、DWHへの取り込みが必要になります。

### 条件1：複数システムをまたぐ問いに答えるため

CRM・Billing・ERPのいずれか1つでは答えられず、複数のシステムのデータを結合しなければ答えられない問いがあります。

```
例：「商談クローズから初回入金までの期間は、顧客セグメント別にどう異なるか？」

  CRM        → 商談クローズ日・顧客セグメント
  Billing    → 初回請求日
  ERP        → 初回入金日

  → 3システムのデータを結合しないと答えられない
```

オペレーショナルシステムはドメインの境界で分断されています。DWHはその境界を越えて問いに答えるための場所です。

### 条件2：時系列の問いに答えるため

オペレーショナルシステムは「今の状態」を管理するために設計されています。契約が更新されれば古い内容は上書きされ、顧客の属性が変わればそれ以前の状態は失われます。

「3ヶ月前の時点でのMRRはいくらだったか」「昨年獲得した顧客コホートの継続率はどう推移しているか」——こうした問いに答えるには、過去の状態のスナップショットが必要です。DWHはこの履歴を保持します。

```
オペレーショナルシステム    DWH
─────────────────          ──────────────────────────
現在の状態のみ保持          月次スナップショットとして
                            過去の状態を蓄積
契約更新 → 旧内容を上書き  更新前後の状態を両方保持
```

### 条件3：派生指標の計算のため

MRR・NRR・コホート継続率などの指標は、単一の業務イベントに対応するレコードとして存在しません。複数のイベントにまたがる計算によって初めて導出される派生指標です。

オペレーショナルシステムにこれらの計算を組み込むと、本来の業務処理とは無関係な集計ロジックが混入し、システムの責務が曖昧になります。派生指標の計算はDWH層の責務です。

---

## 逆に、DWHに乗せる必要がないデータ

上記3条件のいずれも満たさないデータは、DWHに取り込む優先度が低くなります。

**オペレーショナルシステム内で完結する現在状態**　——　「この顧客のサブスクリプションは現在アクティブか」という問いは、BillingシステムがリアルタイムのAPIで答えるべきです。DWHは更新ラグがあるため、業務上のリアルタイム判断には使えません。

**ワークフロー中間状態**　——　「見積のドラフト」「承認待ちの注文」といった一時的なプロセス状態は、ワークフローが完了すれば意味を失います。分析上の価値が低く、DWHに蓄積する必要はありません。

**単一システムで完結する問い**　——　「今月発行した請求書の合計金額」はBillingシステムが直接答えられます。わざわざDWHに取り込まなくても、オペレーショナルシステムのレポート機能で十分な場合があります。

## 本章が扱うテーブル

以上の基準に照らして、本章では次のテーブルを扱います。

| テーブル | 取り込む理由 |
|---|---|
| 請求明細ファクト | Billing単体では商品・顧客・期間を横断した集計が難しい |
| 入金ファクト | 請求と入金のマッチングはBillingとERPをまたぐ |
| 見積明細ファクト | 受注率・割引傾向の分析はCRMとBillingをまたぐ |
| 顧客ディメンション | 複数システムに散在する顧客属性を統合する |
| 商品ディメンション | 商品の分類・属性を分析軸として整備する |
| 売掛金エイジング | 未収金の時系列管理はERPの現在状態では追えない |
| MRRスナップショット | サブスクリプション状態の時系列スナップショット |
| コホートファクト | 顧客獲得時期を起点にした時系列の継続率計算 |
| NRRファクト | 複数月のMRRスナップショットから導出される派生指標 |

契約や注文はオペレーショナルシステム（CRM・Billing）の中で管理される業務オブジェクトです。ただし「契約期間別のチャーン傾向」や「注文から請求までのリードタイム分析」が必要になった場合は、これらも取り込みの対象になります。本章では収益の流れを直接測定するテーブルに絞って説明します。
