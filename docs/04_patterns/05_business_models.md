# 従量課金とプロフェッショナルサービス——固定サブスクリプション以外のビジネスモデル

ここまで説明してきたドメインとパターンは、主に「毎月決まった金額を請求するSaaSサブスクリプション」を想定していました。でも実際のB2B SaaSはそれだけではありません。使った量に応じて料金が変わる従量課金、人の時間を売るプロフェッショナルサービス——これらは業務システムの設計が根本的に変わります。

---

## ビジネスモデルの比較

| | B2B SaaS（SLG） | PLG SaaS | 従量課金型 | プロフェッショナルサービス |
|--|--------------|----------|-----------------|----------------------|
| 入口 | 営業・マーケ | プロダクト（セルフ） | 営業 + セルフ混在 | 営業・提案依頼 |
| 見積・契約 | 必須 | 不要（小規模） | 不要〜任意 | 必須（SOW） |
| 請求金額 | 固定（プラン・シート数） | 固定（プラン） | 変動（使用量） | 工数・マイルストーン |
| 収益の性質 | 経常収益（ARR） | 経常収益 | 変動経常収益 | 非経常収益（案件ごと） |
| 解約の見え方 | 更新拒否で明確 | 随時解約 | 使用量ゼロで暗黙的 | プロジェクト完了で終了 |
| 主要指標 | ARR・NRR | コンバージョン率・DAU | 使用量・MRR | 稼働率・プロジェクト利益率 |

---

## 従量課金型（Usage-based SaaS）

使った量に応じて料金が決まるモデルです。Twilio・AWS・Snowflake・SendGrid などが代表例です。

### 固有のドメイン

固定サブスクリプション型にはない「使用量記録」が登場します。

| ドメイン | 役割 |
|---------|------|
| 使用量記録（Usage Record） | 誰が・いつ・何を・どれだけ使ったかのリアルタイム記録 |
| メータリング | 使用量を正確に計測・集計するための仕組み |

使用量記録の最小構成：

| 項目 | 説明 |
|------|------|
| 顧客 ID | 誰の使用量か |
| 商品 ID | 何を使ったか（APIコール・GB・送信数など） |
| 数量 | いくつ使ったか |
| タイムスタンプ | いつ使ったか |

### 業務フロー

```
請求期間の開始（例：月初）
  ↓
使用量の発生・記録（リアルタイム）
  ↓
請求期間の終了（例：月末）
  ↓
使用量の集計・確定
  ↓
請求書の生成
  ↓
入金（自動決済）
  ↓
次の請求期間へ
```

固定サブスクリプションと異なり「契約更新」というイベントが発生しないことが多いです。使い続ける限り課金が続きます。

### コミットメント付き従量課金

エンタープライズ顧客向けには「最低○○円分は使う」という最低消費額（コミット）を約束する代わりに単価を下げる契約があります。

```
コミット契約（例：年間100万円分）
  ↓ 月次で使用量を消化
  ↓ 消化しきれなかった場合 → 残額を一括請求（または失効）
  ↓ コミット超過の場合 → 超過分を従量で追加請求
```

### 固有のリスク

**収益の予測が難しい**
使用量は月によって変動します。固定サブスクリプションと比べて売上予測が困難です。

**メータリング基盤が必要**
使用量を正確に計測・記録するシステムが、技術的にも運用的にも複雑になりやすいです。

**暗黙的なチャーン**
顧客が使用量をゼロにしても解約通知が来ません。使用量の急減がチャーンのサインになります。

## プロフェッショナルサービス

専門的な知識・技術を持つ人が顧客の課題を解決するために提供するサービスです。コンサルティング・システム導入支援・カスタム開発・研修などが該当します。SaaSとは異なり「人の時間」を売ります。SaaS企業がソフトウェアの導入支援として提供するケースもあります。

### 固有のドメイン

SaaSモデルには存在しないドメインが登場します。

| ドメイン | 役割 |
|---------|------|
| プロジェクト（Project） | 作業の管理単位 |
| マイルストーン（Milestone） | 請求のトリガーとなる節目（「設計完了」「本番稼働」など） |
| 工数記録（Timesheet） | 誰がいつ何時間作業したか（工数ベース請求の場合） |
| SOW（Statement of Work） | 契約の主体。作業範囲・成果物・金額を定める作業指示書 |

### 契約タイプ

| タイプ | 説明 | リスク負担 |
|-------|------|----------|
| 固定価格（Fixed Price） | 作業量に関係なく合意金額を受け取ります | 売り手が負います |
| 工数精算（Time & Materials） | 実際にかかった工数を請求します | 買い手が負います |

どちらを選ぶかで、見積・請求の仕組みが大きく変わります。

### 請求パターン

| パターン | タイミング |
|---------|----------|
| 前払い | 契約締結時（小規模・短期案件） |
| マイルストーン請求 | 各フェーズ完了時（中〜大規模案件） |
| 月次工数請求 | 毎月末（準委任契約） |
| 完了払い | 全作業完了後（売り手にリスクが集中します） |

### 固有の指標とリスク

**稼働率（Utilization Rate）**
コンサルタント・エンジニアが請求可能な作業に使った時間の割合です。稼働率が下がると人件費を回収できません。

**スコープクリープ**
顧客の要望が追加され続け、当初の契約範囲を超えた作業が発生することです（Change Orderが必要）。見積時の作業範囲の明確化と変更管理プロセスが重要です。

**収益認識**
成果物の納品・検収完了をもって売上を計上します。プロジェクト遅延が収益認識の遅れに直結します（サブスクリプションの月次認識とは根本的に異なります）。
