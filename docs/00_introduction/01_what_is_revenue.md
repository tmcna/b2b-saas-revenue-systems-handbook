# 売上とは何か——職種によって変わる「当たり前」の定義

「今月の売上はいくらでしたか?」

この質問を、同じ会社の5人に聞いてみると面白いことが起きます。

- **営業部長**：「3,000万です。5件クローズできました」
- **CFO**：「今月計上できるのは2,100万ですね」
- **経理担当**：「財務諸表上は1,800万になります」
- **RevOps担当**：「MRRベースだと月次200万の積み上げがありました」
- **カスタマーサクセス**：「既存顧客からの拡張が800万ありましたよ」

全員が正直に答えています。しかし誰も同じ数字を言っていない。

これは数字の計算ミスではありません。コミュニケーションの失敗でもありません。**それぞれが「自分の仕事における売上」を正確に語っているのです。**

---

## 職種別の「売上」——5つの異なる意味

### 営業が見る売上：Bookings（受注）

営業にとっての売上とは、「今月クローズした案件の合計金額」です。契約書に署名が入った瞬間、その金額は「売上」として記録されます。

年間契約で3,000万円の案件をクローズすれば、それは今月の売上3,000万円です。来月以降にサービスを提供するかどうかは関係ない。**合意した価値が確定した、それが重要なのです。**

この概念を英語では **Bookings（ブッキングス）** と呼びます。受注額、あるいは「予約」された収益です。

### CFOが見る売上：Revenue（収益認識）

CFOや財務担当者が「売上」という言葉を使うとき、それは会計基準に従って**すでにサービスを提供した分だけ**を意味します。

3,000万円の年間契約を締結しても、今月提供したサービスが1ヶ月分であれば、今月の売上は250万円（3,000万 ÷ 12ヶ月）です。残りの2,750万円はまだ「前受収益（Deferred Revenue）」であり、提供が完了してはじめて売上に変わります。

これを **Revenue Recognition（収益認識）** と言います。国際会計基準（IFRS 15）や米国会計基準（ASC 606）によって厳密なルールが定められており、「いつ売上を計上できるか」は会計的な判断が必要な問題です。

### 経理が見る売上：財務諸表上の売上高

財務諸表（損益計算書）に記載される売上高は、会計期間中に認識した Revenue の合計です。CFOの視点と近いですが、経理担当者が扱うのは勘定科目・仕訳・期間配分といったより細かい会計処理です。

同じ「売上」でも、消費税を含むか含まないか、割引の処理方法、返金・クレジットノートの扱いによって数字が変わります。

### RevOpsが見る売上：ARR/MRR

RevOps（Revenue Operations）担当者やアナリストが最もよく使う指標は **MRR（Monthly Recurring Revenue）** と **ARR（Annual Recurring Revenue）** です。

MRRは「今この瞬間、毎月入ってくるはずの契約額の合計」です。新規契約が増えれば New MRR が上がり、解約が起きれば Churned MRR として下がり、既存顧客がプランをアップグレードすれば Expansion MRR として記録されます。

MRRは会計上の概念ではありません。法律的な定義もない。**しかし SaaS ビジネスの健全性を測るうえで最も重要な指標のひとつです。**

| 指標 | 意味 | 会計上の位置づけ |
|------|------|----------------|
| Bookings | 受注額（契約締結時点） | 会計上の売上ではない |
| Revenue | 収益認識済みの売上 | P&L に計上される |
| ARR/MRR | 年次・月次の定期収益 | 会計上の定義なし |

### カスタマーサクセスが見る売上：NRR（Net Revenue Retention）

カスタマーサクセスチームが重視するのは **NRR（Net Revenue Retention）**、つまり「既存顧客から今月どれだけの収益が残り、さらに増えたか」です。

解約・ダウングレードで減った分と、アップグレード・追加契約で増えた分を合算し、前月比で何%かを示します。NRR 110% であれば、既存顧客だけで毎月10%ずつ収益が成長していることを意味します。

CS担当者にとっての「売上を増やす」とは、新規顧客を獲得することではなく、**既存顧客を拡張・維持すること**です。

## なぜ同じ言葉が違う意味を持つのか

これは定義の混乱ではありません。**それぞれの職種が、まったく異なる時点の異なるイベントを追跡しているのです。**

```
時間軸 →

  商談開始     契約締結     サービス開始    請求発行     入金
    |            |             |             |           |
    |      Bookings↑           |             |           |
    |            |    Revenue を少しずつ認識  |           |
    |            |             |    MRR に反映          |
    |            |             |             |   Cash in |
```

営業は「合意した瞬間」を見ています。財務は「提供が完了した分」を見ています。RevOpsは「継続中の契約の積み上がり」を見ています。経理は「仕訳として確定した金額」を見ています。

全員が同じビジネスを見ているのに、着目している時点とイベントが違います。

## 「売上」の曖昧さがシステムを複雑にする

この曖昧さは、業務システムの設計に直接影響します。

たとえば「売上をリアルタイムで確認したい」という要件を受け取ったとき、それが **Bookings なのか Revenue なのか MRR なのか** によって、必要なデータソースも計算ロジックも、さらにはどのシステムが正本（Source of Truth）を持つべきかも変わります。

- Bookings を知りたいなら → CRMの商談データ
- Revenue を知りたいなら → ERPの売上計上データ
- MRR を知りたいなら → サブスクリプション管理システムのデータ

「売上のダッシュボードをつくってほしい」という依頼が炎上しやすいのは、依頼者と実装者が違う「売上」を想定しているからです。

---

## この本で扱う「売上」

本書では「売上」を特定の会計的定義に限定しません。むしろ、**Bookings・Revenue・MRR/ARR・Cash の4つが、業務の流れの中でどう発生し、どのシステムに記録され、どう連鎖するか** を追います。

次の章では、CRM・Billing・ERP という3つのシステムがなぜデータを統合できないのかを説明します。「売上がシステムによってずれる」問題の構造的な理由が、そこにあります。
