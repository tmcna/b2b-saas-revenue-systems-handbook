# 契約ドメイン——署名した瞬間に変わる「不可逆性の境界」

見積書は何度でも修正できます。でも、契約書に署名した瞬間に状況は変わります。双方に法的義務が生まれ、「取り消せる状態」から「取り消せない状態」へと移行します。この転換点を正確に記録することが、契約ドメインの本質的な役割です。

契約ドメインは「コミットメントエンティティ（Commitment Entity）」を管理します。合意が確定した瞬間の記録であり、変更には合意し直しが必要という特徴を持ちます。

---

## なぜ契約を変更できないのか

契約が変更されるとき（アップグレード・ダウングレード）、既存の契約は変更されず、「改訂（Amendment）」として別のレコードが作られます。これは追記のみ（Append-only）の原則に従った設計です。

「合意した内容」の完全な履歴が残ることが、紛争時の根拠になります（民事訴訟法上の証拠保全の観点）。電子データで締結・保存した契約書は電子帳簿保存法の適用対象となり、訂正・削除の事実が確認できる状態での保存が求められます。過去の合意内容を後から書き換えることができれば、どちらが正しい記録かわからなくなります。改訂履歴として新しいレコードを積み上げることで、「いつ・何を・どう変えたか」が追跡可能になります。

---

## 主なエンティティ

**契約（Contract）**
取引全体の合意を表します。契約期間・支払条件・ステータスなどを持ちます。

**契約明細（Contract Line Item）**
契約の中の個々の商品行です。商品・数量・合意単価・契約期間を持ちます。

**改訂（Amendment）**
契約期間中に内容が変更されたときの記録です。アップグレード・ダウングレード・数量変更などが該当します。

## ライフサイクル

```
下書き（Draft）
  ↓
署名待ち（Pending Signature）
  ↓
有効（Active）
  ↓
更新（Renewed）→ 新しい契約期間へ
  または
解約申請（Pending Cancellation）
  ↓
終了（Terminated / Expired）
```

「Expired」は契約期間が自然に終了したケース、「Terminated」は途中解約のケースで使い分けることがあります。

## 識別子

| エンティティ | 識別子の例 | 注意点 |
|------------|----------|--------|
| 契約 | 契約番号（例：C-2024-0001） | 顧客に提示する番号 |
| 改訂 | 契約番号 + 改訂番号（例：C-2024-0001-A1） | 何度目の改訂かを追跡できる形式が望ましいです |

---

## 契約期間と請求サイクル

契約には「契約期間」と「請求サイクル」の2つの時間軸があります。

| 概念 | 説明 | 例 |
|------|------|-----|
| 契約期間（Term） | 契約が有効な期間 | 2024/4/1 〜 2025/3/31（1年間） |
| 請求サイクル | 請求書を発行する間隔 | 毎月 / 毎年 |

年間契約でも月払いにすることがあります。この場合、契約期間は1年ですが、請求は12回発生します。

## 他のドメインとの関係

```
契約
  ├── 顧客（アカウント）← 顧客ドメイン
  ├── 見積 ← 見積ドメイン（契約の元になった見積）
  ├── 契約明細
  │     └── 商品 ← 商品ドメイン
  ├── サブスクリプション → サブスクリプションドメイン
  └── 請求書 → 請求書ドメイン
```

契約は見積から生まれ、サブスクリプション・請求書を生み出す、業務フローの中心に位置します。

## 正本となるシステム

**CRM** または専用の**契約管理システム**が正本になります。SalesforceのContractsオブジェクト、HubSpotのDealsなど、CRMに契約管理が内包されているケースもあります。

請求システムにも契約情報が存在しますが、請求システムが持つのは「請求に必要な情報のコピー」であり、契約の正本はCRM側が持つ設計が一般的です。

## 契約の変更（Amendment）

サブスクリプションの途中で契約内容が変わることがあります。

| 変更の種類 | 説明 |
|-----------|------|
| アップグレード | 上位プランへの変更・数量の増加 |
| ダウングレード | 下位プランへの変更・数量の減少 |
| 追加 | 新しい商品の追加 |
| 解約申請 | 契約終了の申し込み |

変更が発生した場合、**日割り計算（Proration）** が必要になることがあります。契約期間の途中でアップグレードした場合、残り期間分の差額を請求します。

---

## よくある課題

**更新管理の抜け漏れ**
契約更新のタイミングを逃すと、意図しない契約終了が発生します。更新予定日の自動アラートが必要です。

**改訂の複雑化**
複数回の改訂が積み重なると、「現在の契約内容」が何なのか把握しにくくなります。常に現在の有効な契約内容を参照できる設計が重要です。

**日割り計算のロジック**
契約途中の変更に伴う日割り計算は、実装が複雑になりやすい部分です。どのタイミングで・どのように計算するかを、仕様として明確に定めておく必要があります。
