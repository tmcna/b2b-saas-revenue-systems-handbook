# 注文——「合意」から「実行」へ、契約と注文が分かれる理由

契約ドメインと注文ドメインは、混同されやすいドメインです。どちらも「受注した内容」を管理するように見えます。でも役割は根本的に異なります。契約は「何を約束したか」の法的記録であり、注文は「その約束を今すぐ実行せよ」という指示です。

注文ドメインは「履行オブジェクト（Fulfillment Object）」を管理します。合意した内容を実際に「動かす」ための指示を記録する役割を持ちます。

---

## なぜ契約と注文を分けるのか

1つの契約から複数の注文が生まれます。月次サブスクリプションなら毎月1回、注文が自動生成されます。アップグレード・数量変更・一時停止が起きると、新しい注文が発生します。

契約に変更履歴を直接書き込むのではなく、「注文」として別に記録することで、「現在の合意（契約）」と「実行の履歴（注文）」が分離できます。契約は「何を約束したか」の不変の記録として残り、注文は「いつ・何を実行したか」の積み上げ記録になります。

---

## 主なエンティティ

**注文（Order）**
実行指示全体を表します。注文日・ステータス・請求先・合計金額などを持ちます。

**注文明細（Order Line Item）**
注文の中の個々の商品行です。商品・数量・単価・小計を持ちます。

---

## ライフサイクル

```
下書き（Draft）
  ↓
確認済み（Confirmed）
  ↓
処理中（In Progress）
  └── サービス開始 / プロビジョニング
  ↓
完了（Fulfilled）
  ↓
（請求書ドメインへ）
```

キャンセルは「確認済み」以前であれば可能ですが、処理中・完了後のキャンセルは返金・取り消しの処理が必要になります。

---

## 識別子

| エンティティ | 識別子の例 | 注意点 |
|------------|----------|--------|
| 注文 | 注文番号（例：O-2024-0001） | 顧客に提示する番号 |
| 注文明細 | 注文ID + 行番号 | — |

B2B取引では、買い手が「注文書番号（PO番号）」を発行し、売り手の注文番号と紐づけます。請求書にPO番号を記載することを求められることがあります。

---

## 契約と注文の役割の違い

| | 契約 | 注文 |
|--|------|------|
| 目的 | 合意の記録 | 実行の指示 |
| 発生タイミング | 署名時 | サービス開始・更新・変更時 |
| 数 | 1件の取引に1つ | 1件の契約から複数発生しうる |

サブスクリプションでは、1つの契約から毎月（または毎年）注文が自動生成されます。

```
契約（1件）
  ├── 注文（初月）→ 請求書
  ├── 注文（2ヶ月目）→ 請求書
  └── 注文（3ヶ月目）→ 請求書
```

## 他のドメインとの関係

```
注文
  ├── 顧客（アカウント）← 顧客ドメイン
  ├── 契約 ← 契約ドメイン
  ├── 注文明細
  │     └── 商品 ← 商品ドメイン
  └── 請求書 → 請求書ドメイン
```

## 正本となるシステム

注文ドメインの正本は、ビジネスモデルによって異なります。

| ビジネスモデル | 正本になりやすいシステム |
|-------------|----------------------|
| SaaS（サブスクリプション） | 請求システム（Billing）が注文を自動生成します |
| プロジェクト型 | CRM / 契約管理システム |
| ECコマース | ECプラットフォーム |

SaaSでは、契約の開始・更新・変更イベントをトリガーに、請求システムが注文を自動生成することが多いです。

---

## よくある課題

**契約変更時の注文の扱い**
アップグレードや数量変更が発生すると、残りの契約期間に対して差額の注文を新規に作るか、既存の注文を修正するかの設計判断が必要です。

**PO番号の管理**
大企業の顧客は、請求書にPO番号の記載を求めることがあります。PO番号を注文と紐づけ、請求書に自動で引き継ぐ仕組みが必要です。

**注文と請求のタイミングのずれ**
注文が完了しても、請求書の発行タイミングは別に管理されます。注文の完了ステータスと請求書の発行ステータスを混同しないよう注意が必要です。
