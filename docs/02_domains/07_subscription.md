# サブスクリプションドメイン——「今まさに提供中」の状態を管理する

SaaSビジネスの核心は「継続的なサービス提供」にあります。契約は「何を約束したか」の静的な記録ですが、サービスは毎日動いています。アップグレード・ダウングレード・一時停止・解約——これらの変化を動的に追跡するのが、サブスクリプションドメインの役割です。

サブスクリプションドメインは「履行エンティティ（Fulfillment Entity）」を管理します。注文ドメインと同じ型ですが、注文が「実行の指示」であるのに対し、サブスクリプションは「今どの状態にあるか」の継続的な管理を担います。

---

## なぜ契約とサブスクリプションを分けるのか

契約は「法務・営業の視点」——署名した合意内容の不変の記録です。サブスクリプションは「プロダクト・請求の視点」——今提供しているサービスの状態です。

大規模なSaaSでは、1つの契約から複数のサブスクリプションが生まれることがあります（複数プロダクト・複数拠点など）。この多対多の関係を扱うには分離が必要です。

また、Active → Paused → Cancelled → Renewed と状態が変化し、それぞれの状態変化が請求イベントのトリガーになります。この動的な状態管理を、変更不可の契約エンティティには持たせられません。

---

## 主なエンティティ

**サブスクリプション（Subscription）**
顧客への継続的なサービス提供を表します。現在のプラン・数量・開始日・終了日・ステータスを持ちます。

**サブスクリプション明細（Subscription Line Item）**
サブスクリプションの中の個々の商品行です。複数の商品をまとめて1つのサブスクリプションで管理します。

**サブスクリプションイベント（Subscription Event）**
サブスクリプションに起きた変化の記録です。開始・アップグレード・ダウングレード・解約などのイベントを履歴として残します。

## ライフサイクル

```
トライアル（Trial）※無料試用期間がある場合
  ↓
有効（Active）
  ├── アップグレード → 新しいサブスクリプション明細へ
  ├── ダウングレード → 新しいサブスクリプション明細へ
  └── 数量変更 → 明細の数量を更新
  ↓
更新待ち（Pending Renewal）
  ↓
更新（Renewed）→ 次の契約期間へ
  または
解約申請（Pending Cancellation）
  ↓
解約済み（Cancelled）
```

## 契約ドメインとの違い

似ているようで役割が異なります。

| | 契約 | サブスクリプション |
|--|------|-----------------|
| 目的 | 合意の法的記録 | サービス提供状態の管理 |
| 変更 | 改訂（Amendment）として記録 | ステータスと明細を更新 |
| 視点 | 法務・営業 | プロダクト・請求 |
| 終了後 | アーカイブとして残します | 解約履歴として残します |

システムによっては契約とサブスクリプションが同一のエンティティで管理されることもありますが、大規模なSaaSでは分離して設計されることが多いです。

---

## 他のドメインとの関係

```
サブスクリプション
  ├── 顧客（アカウント）← 顧客ドメイン
  ├── 契約 ← 契約ドメイン
  ├── サブスクリプション明細
  │     └── 商品 ← 商品ドメイン
  └── 請求書 → 請求書ドメイン
        └── （請求サイクルごとに自動生成）
```

サブスクリプションの状態が、請求書の自動生成をトリガーします。アクティブなサブスクリプションがある限り、請求サイクルのたびに請求書が生成されます。

## 正本となるシステム

**請求システム（Billing）** がサブスクリプションの正本を持つことが多いです。Stripe Billing、Zuora、Chargebee などの専用の請求システムは、サブスクリプション管理を中心機能として持っています。

CRMが正本を持つ設計もありますが、請求システムとの二重管理になりやすいため、どちらかに集約するか、明確な同期ルールを設ける必要があります。

---

## 重要な指標との関係

サブスクリプションドメインのデータは、ビジネスの健全性を測る指標の源泉になります。

| 指標 | 計算の元データ |
|------|-------------|
| MRR（月次経常収益） | アクティブなサブスクリプション明細の月次換算金額の合計 |
| ARR（年次経常収益） | MRR × 12 |
| チャーンレート（解約率） | 解約したサブスクリプションの件数・金額の割合 |
| 拡張収益（Expansion Revenue） | アップグレード・追加購入による収益増加分 |

## よくある課題

**トライアルから有料への転換管理**
無料トライアルが終了したタイミングで、自動的に有料プランに移行するか、手動で確認するかの設計が必要です。

**複数サブスクリプションの管理**
1顧客が複数のサブスクリプションを持つ場合（複数プロダクト・複数拠点など）、請求を統合するか分けるかの判断が必要です。

**解約申請と即時解約の違い**
「今すぐ解約」と「契約期間終了時に解約」では、サービス停止のタイミングと返金の扱いが異なります。解約ポリシーをシステムに正確に反映する必要があります。
