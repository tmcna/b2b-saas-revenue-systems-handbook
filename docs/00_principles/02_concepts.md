# 業務システムを語るための基礎概念

ここでは、業務システムを理解・設計するうえで繰り返し登場する5つの概念を整理します。「正本とは何か」「マスターデータとトランザクションデータの違いは」「なぜ部門によって言葉の意味が変わるのか」——これらを言語化しておくと、ドメインごとの設計判断の理由が見えやすくなります。

---

## 正本（System of Record）

あるデータの「唯一の正しい情報源」として扱うシステムのことです。英語では System of Record（SoR）と呼びます。本ドキュメントでは SoR の訳語として「正本」を使いますが、公正証書や公文書における法的用語「正本」（謄本・抄本に対する概念）とは異なります。

同じデータ（例：顧客の会社名）が CRM・請求システム・ERP に分散している場合、どれが正しいか決まっていないと、修正のたびに複数箇所を直す必要があります。最終的にシステム間で値がずれていきます。

**更新の方向は一方向に保ちます：** 正本に対して更新し、他のシステムはそれを受け取ります。逆方向の更新は整合性を壊します。

```
CRM（正本）→ 同期 → 請求システム（コピー）→ 同期 → ERP（コピー）
```

| データ | 正本になりやすいシステム |
|--------|----------------------|
| 顧客情報 | CRM |
| 商品・価格 | CPQ / 商品管理システム |
| 契約情報 | CRM / 契約管理システム |
| 請求・入金 | 請求システム |
| 会計仕訳 | ERP |

正本が決まっていないと、どのシステムの情報が最新かわからず、レポートを作るたびに集計結果がシステムによって異なります。データを修正するとき、どこを直せばよいかもわかりません。

---

## マスターデータ

業務の「基準」となる、変化が少ない基本情報のことです。複数の業務プロセスから参照されます。

| マスターデータ（参照される側） | トランザクションデータ（参照する側） |
|--------------------------|--------------------------------|
| 顧客・商品・価格表 | 見積・注文・請求書・入金 |
| 変化が少ない | 日々積み上がります |

トランザクションデータは必ずマスターデータを参照します（「誰への請求書か」→ 顧客マスター）。

**分散による問題：** 複数のシステムにマスターデータが分散すると、同一顧客が重複登録される・システムによって顧客名が異なる・どれが最新かわからない、という問題が起きます。まず正本を決めることが現実的な第一歩です。統一管理の取り組みを MDM（マスターデータ管理、Master Data Management）と呼びます。

---

## 境界づけられたコンテキスト

ある言葉や概念が「同じ意味で通用する範囲」のことです。英語では Bounded Context と呼びます（ドメイン駆動設計の概念です）。

同じ「顧客」という言葉でも、部門・システムによって意味が違います：

| システム | 「顧客」が指すもの |
|---------|----------------|
| 営業（CRM） | 商談相手の会社 |
| 請求システム | 請求書の送付先 |
| 会計（ERP） | 売掛金の管理対象 |

コンテキストをまたいでデータをやり取りするときは**変換**が必要になります。CRM の「商談」を請求システムの「注文」に変換するような変換ルールを曖昧にしたまま連携すると、データ不整合が生じます。

B2B SaaS の業務システムが分かれるコンテキスト：

| コンテキスト | 代表システム |
|------------|-----------|
| 顧客管理 | CRM |
| 見積・価格 | CPQ |
| 契約管理 | CRM / 契約管理システム |
| 請求・入金 | 請求システム |
| 会計 | ERP |

---

## イベント

業務の中で起きた「出来事」の記録です。「契約が締結された」「請求書が発行された」など、過去に起きた事実を指します。イベントは取り消せません——起きた事実として記録され続けます。

業務システムは「現在の状態」だけでなく「何が起きたか」の履歴を必要とします。契約の状態が「有効」でも、その契約がどのような経緯で今の状態になったかはイベント履歴を見ないとわかりません。

**イベントの最小構成：**

| 項目 | 説明 |
|------|------|
| イベント名 | 何が起きたか（例：契約締結、請求書発行） |
| 発生日時 | いつ |
| 対象 | どのデータに関するイベントか（例：契約 ID） |
| 発生者 | 誰が・何が起こしたか |

あるイベントが次の業務プロセスのトリガーになります：

```
「契約締結」イベント
  → サブスクリプション開始処理
  → ウェルカムメール送信
  → 初回請求書の発行
```

複数のシステムをつなぐとき、イベントを起点に各システムが必要な処理を行う設計を「イベント駆動（Event-Driven）」と呼びます。

---

## 業務プロセス

ある目的を達成するための一連の作業の流れです。例えば「顧客に商品を売って代金を受け取る」ために、見積作成 → 契約 → 請求 → 入金確認、という流れが発生します。

代表的な業務プロセスの名称と始点：

| プロセス名 | 始点 | 主な利用者 |
|-----------|------|----------|
| Lead-to-Cash | リード獲得 | マーケティング・RevOps |
| Quote-to-Cash | 見積作成 | 営業・CPQ 担当 |
| Order-to-Cash | 受注（注文） | オペレーション・財務 |

これら3つは独立したプロセスではなく入れ子の関係で、終点はすべて同じ（入金）です。詳細はプロセス編で解説します。
