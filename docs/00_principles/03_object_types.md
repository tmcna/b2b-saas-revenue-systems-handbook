# 見積書と請求書はなぜ別のシステムで管理するのか——業務オブジェクトの4つの型

前回は、業務システムが統合できない3つの構造的な理由を見てきました。今回は、その理解をさらに深める「型分類」を紹介します。

業務システムに登場するオブジェクト（見積・契約・請求書など）は、種類が多く見えます。でも実は、**4つの型**に分類できます。これはKimball ToolkitのFact Table型分類（Transaction・Periodic Snapshot・Accumulating Snapshot）に相当する、業務オブジェクト版の基本的な分類体系です。

この型を理解すると、任意の業務オブジェクトの性質——可変か不変か、いつ確定するか、どのシステムが正本を持つべきか——をその都度考えずに導き出せるようになります。

---

## 型1：交渉オブジェクト（Negotiation Object）

コミットメント前の段階を記録するオブジェクトです。

- 何度でも修正できます
- 破棄しても法的・財務的な影響はありません
- 「最終的にどうなるか」はまだ確定していません

| オブジェクト | 説明 |
|-----------|------|
| 商談（Opportunity） | 取引が成立するかどうか不明な段階 |
| 見積書（Quote） | 提示した条件。顧客が拒否すれば無効になります |

**所有ドメイン：** 営業ドメイン（CRM / CPQ）

---

## 型2：コミットメントオブジェクト（Commitment Object）

合意が確定した瞬間を記録するオブジェクトです。

- 作成後は原則変更できません（変更には合意し直しが必要です）
- 双方が義務を負います
- 「不可逆性の境界」を形成します

| オブジェクト | 説明 |
|-----------|------|
| 契約（Contract） | 双方が署名し、義務が発生した時点の合意内容 |

**所有ドメイン：** 契約管理ドメイン（CRM / 契約管理システム）

---

## 型3：履行オブジェクト（Fulfillment Object）

合意した内容を実行中であることを表すオブジェクトです。

- 状態機械として機能します（Active → Paused → Cancelled など）
- 期間を持ちます（開始・終了・更新サイクル）
- 「今、何が提供されているか」の正本です

| オブジェクト | 説明 |
|-----------|------|
| 注文（Order） | 履行の開始を指示するオブジェクト |
| サブスクリプション（Subscription） | 継続的な履行の状態を管理するオブジェクト |

**所有ドメイン：** 履行ドメイン（Billing / サブスクリプション管理）

---

## 型4：決済オブジェクト（Settlement Object）

履行の対価を確定させるオブジェクトです。

- 追記のみ（Append-only）です。発行後は変更できません
- 法的・会計的に不可逆です
- 誤りは打ち消し（クレジットノート、差し戻し）で対処します

| オブジェクト | 説明 |
|-----------|------|
| 請求書（Invoice） | 代金請求の法的記録 |
| 入金（Payment） | 代金受領の確定記録 |

**所有ドメイン：** 財務ドメイン（Billing / ERP）

---

## 型が変わる境界こそが「システムの境界」

業務プロセスは、この4つの型を順に通過します。型が変わる瞬間が「不可逆性の境界」であり、そこでシステムも変わります。

```
[営業ドメイン]                    [財務ドメイン]
Negotiation → Commitment → Fulfillment → Settlement
   見積書  →     契約   →     注文    →   請求書
                  ↑                          ↑
           ここで可逆から               ここで履行から
           不可逆へ変わります           対価確定へ変わります
```

**見積書と請求書が分離される理由：** 見積書（Quote）はNegotiation Object（破棄可能）で、請求書（Invoice）はSettlement Object（取消不可）です。両者はオブジェクトの型が根本的に異なります。これを同一のデータモデルに収めることは、「まだ合意していない価格」と「法的に確定した請求」を混同することになります。

---

## 新しいオブジェクトが登場したときの判定方法

新しい業務オブジェクト（例：「Proposal」「Work Order」「Credit Memo」）が登場したとき、以下の問いに答えるだけで型が決まります。

1. コミットメント前か後か？ → 前なら **Negotiation**
2. 双方の合意を記録しているか？ → Yes なら **Commitment**
3. 実行中の状態を追跡しているか？ → Yes なら **Fulfillment**
4. 代金・金額の確定を記録しているか？ → Yes なら **Settlement**

型が決まれば、更新モデル（可変 / 追記のみ）・正本システム・ライフサイクルのパターンが自動的に決まります。
