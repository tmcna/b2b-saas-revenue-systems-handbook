# 【補足編】マルチエンティティ・多通貨——複数法人と複数通貨が生み出す構造的な複雑さ

単一の法人・単一の通貨で事業を運営している間は見えない問題があります。日本法人と米国法人を持ち、顧客への請求は USD と JPY が混在し、連結決算は JPY でまとめる——この状態になると、Billing・ERP・DWH のすべてに設計上の判断が必要になります。

---

## マルチエンティティとは

**エンティティ（Entity）** とは、会計上独立した法人格を持つ組織単位です。日本法人・米国法人・シンガポール法人はそれぞれ独立したエンティティです。

マルチエンティティ構成では、以下の問いに答える必要があります。

- どの法人が顧客と契約するか（契約主体）
- どの法人が請求書を発行するか（請求主体）
- 各法人でどの税制が適用されるか
- 複数法人の財務諸表をどう連結するか

---

## 通貨の3つの概念

多通貨取引を扱う際、3種類の通貨を区別する必要があります。

| 通貨 | 定義 | 例 |
|---|---|---|
| 請求通貨（Transaction Currency） | 顧客へ請求する通貨 | 米国顧客には USD、日本顧客には JPY |
| 機能通貨（Functional Currency） | 各法人の会計上の基準通貨 | 米国法人は USD、日本法人は JPY |
| 報告通貨（Reporting Currency） | 連結決算の通貨 | 親会社が日本なら JPY |

請求通貨と機能通貨が一致しない場合（例：日本法人が USD で請求する場合）、入金時の為替レートと請求時の為替レートの差が「**為替差損益**」として発生します。

---

## 為替レートの3種類

財務会計では、取引の種類によって使用する為替レートが異なります。

| レートの種類 | 使用タイミング | 例 |
|---|---|---|
| 取引日レート（スポットレート） | 売上・費用の仕訳計上時 | 請求書発行日のレート |
| 決済日レート | 入金・支払い時の換算 | 実際の入金日のレート |
| 期末レート | 貸借対照表の外貨建て残高の換算 | 月末・四半期末のレート |
| 期中平均レート | 損益計算書の換算（連結） | 月次・四半期の平均レート |

この複数レートの使い分けが、為替差損益・換算差額を生み出します。

---

## Billing への影響

**請求通貨の管理**

顧客ごとに請求通貨を設定し、その通貨で請求書を発行します。Billing システムは複数通貨の請求書生成と、通貨ごとの MRR 集計に対応する必要があります。

**通貨の固定**

年間契約を USD で締結した場合、為替レートが変動しても請求金額（USD）は変わりません。円安が進んでも顧客への請求は USD のままです。顧客にとっての安定性と、売り手側の収益の安定性はトレードオフになります。

**為替差損益の発生タイミング**

```
請求書発行（1ドル = 140円で計上）
  ↓
入金（1ドル = 150円で換算）
  ↓
為替差益 = (150 − 140) × 請求USD金額
```

Billing は USD の金額のみ管理し、ERP が JPY 換算と為替差損益の仕訳を担います。

---

## ERP（General Ledger）への影響

**各法人に独立した元帳**

マルチエンティティ構成では、各法人がそれぞれ独立した総勘定元帳（GL）を持ちます。日本法人の売上と米国法人の売上は別々に計上されます。

**内部取引（Intercompany Transaction）の消去**

グループ内の取引（例：日本法人が米国法人にサービスを提供する）は、連結時に消去する必要があります。これを「内部取引消去」と呼びます。消去対象の取引を識別するためのフラグ・勘定科目の設計が必要です。

**連結決算**

各法人の財務諸表を報告通貨（JPY）に換算して合算します。この換算では期中平均レートと期末レートを使い分けます。換算差額（**換算調整額**）は貸借対照表の純資産に計上されます。

---

## DWH への影響

分析基盤では、複数通貨の収益を比較可能にするための設計が必要です。

**通貨統一の方法**

| アプローチ | 内容 | 注意点 |
|---|---|---|
| 固定レート換算 | 年度初めのレートで全件換算 | 計算が単純。実際の損益とずれる |
| 日次レートで換算 | 各取引日のレートを使用 | 正確だが為替テーブルの整備が必要 |
| 原通貨のまま保持 | 換算せず元の通貨で管理 | 通貨をまたいだ集計ができない |

通常は「原通貨と換算後通貨の両方を保持する」設計が最も汎用性が高く、分析側で必要に応じてどちらも使えます。

**為替レートのディメンション**

```sql
-- 為替レートテーブルの例
dim_exchange_rate (
  rate_date         DATE,
  from_currency     VARCHAR,  -- 'USD'
  to_currency       VARCHAR,  -- 'JPY'
  rate              DECIMAL,
  rate_type         VARCHAR   -- 'SPOT' / 'MONTH_END' / 'MONTH_AVG'
)
```

## エンティティ・通貨の組み合わせによる設計の複雑さ

マルチエンティティ × 多通貨の組み合わせは、以下の問いを生み出します。

- 顧客は日本法人と契約しているが、請求は USD でよいか
- 米国法人が JPY で請求する場合の税務上の扱いは
- 同一顧客が複数法人と個別に契約している場合、グループ単位の売上をどう集計するか

これらは個別の法的・税務的判断が必要であり、システム設計の前に契約形態・税務方針を決定する必要があります。

---

## よくある課題

**為替差損益の追跡**
入金が遅れるほど、計上時と入金時のレート差が広がります。月末処理で未収売掛金の換算差額を調整する仕訳（換算替え）が必要になります。

**グループ顧客の名寄せ**
日本親会社・米国子会社がそれぞれ同一グループのベンダーから個別に契約しているとき、グループ全体の ARR を正しく集計するためには、顧客マスタに「グループ（親組織）」の概念が必要です（[ドメイン編 1 顧客](../01_domains/01_customer.md)参照）。

**通貨ごとの MRR 集計**
USD MRR と JPY MRR を単純に足すことはできません。管理会計での MRR レポートは、換算後の単一通貨で統一するか、通貨別に分けて報告するかを決めておく必要があります。
