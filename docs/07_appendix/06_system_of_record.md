# データ正本マトリックス——どのデータがどのシステムで管理されるか

各ドメインのデータを「正本（System of Record）」として管理するシステムを一覧にまとめます。データを更新するとき・問題が起きたとき・システムを連携するときに「どこが正本か」が判断の起点になります。

---

| 商品・価格表 | ◎ | ○ | — | — | ○ |

| 契約 | ◎ | ○ | — | — | ○ |

| 請求書 | — | ◎ | ○ | — | ○ |

### 凡例

| 記号 | 意味 |
|------|------|
| ◎ | **正本**。このシステムが唯一の更新源。他システムはここから同期する |
| ○ | **参照コピー**。正本から同期された読み取り用のデータ |
| △ | **構成による**。システムの組み合わせや成長フェーズによって正本が変わる |
| — | 管理しない |

---

## △ の詳細

### 注文（Order）

SaaSのサブスクリプションモデルでは、Billingシステムが契約の開始・更新・変更イベントを受けて注文を自動生成するため、Billingが正本になります。プロジェクト型の取引ではCRM/契約管理システムが注文を手動で起票するため、CRMが正本になります。

### 入金（Payment）

BillingとERPのどちらを正本にするかは、組織の会計フローによって異なります。

- **Billingを正本にする場合**：「請求書単位の消込状況」はBillingが管理し、ERPには消込済み情報を連携する
- **ERPを正本にする場合**：Billingで仮消込し、ERPが最終的な売掛金消込の正本を持つ

どちらの場合も、2つのシステムで同じ入金データを管理することになるため、「どちらが正本か」を明確にしないと数字がずれます。

### 収益認識スケジュール（Revenue Recognition Schedule）

正本は採用するツールによって3パターンあります（[収益認識スケジュールページ](../05_dwh/10_fact_revenue_schedule.md#上流システムどこからデータを取るか)参照）。

| パターン | 正本 |
|---------|------|
| Zuora Revenue / NetSuite 等が収益認識モジュールを持つ | Billing または ERP |
| GAAP/IFRS 監査対応でERPが収益認識の正本 | ERP |
| DWH で `fact_invoice_line` から計算（Pattern A） | DWH |

### コミッション（Commission）

規模が小さい段階ではCRM（Salesforce・HubSpotの拡張機能）またはスプレッドシートが正本になります。コミッションの複雑さが増す（クォータ管理・クローバック・複数プランの混在など）につれて、Spiff・CaptivateIQ等のコミッション専用システムに移行します。

## 成長フェーズと正本の変化

どのシステムが正本を持つかは、成長フェーズによって変化します。[第4章](../04_patterns/)で説明したパターンと対応させると以下のようになります。

### Early Stage SLG（スタートアップ初期）

CRMがほぼすべてを担います。請求書もスプレッドシートやCRMの機能で発行します。データの正本が分散していないため、運用はシンプルですが、規模拡大とともに限界が来ます。

```
顧客 / 見積 / 契約 / 請求書 → CRM（またはスプレッドシート）
```

### Startup SLG

CRMとBillingが分離します。Billingがサブスクリプション・請求書・入金の正本を持ち始め、「CRMの商談金額とBillingの請求金額を一致させる」という連携課題が初めて発生します。

```
顧客 / 見積 / 契約     → CRM
サブスクリプション / 請求書 / 入金 → Billing
```

### Enterprise SLG

CPQ・ERP・コミッション専用システムが加わります。正本を持つシステムの数が増え、「どのシステムが何の正本か」の設計が経営上の意思決定に直結する重要な判断になります。

```
顧客 / 見積 / 契約         → CRM / CPQ
商品 / 価格表              → CPQ
サブスクリプション / 請求書 / 入金 → Billing
総勘定元帳（財務会計）      → ERP
収益認識スケジュール         → ERP / Billing
コミッション               → コミッション専用システム
管理会計 / 指標            → DWH
```

### PLG（セルフサーブ）

CRMの役割が軽くなり、BillingとDWHが中心になります。見込み客の管理よりも、プロダクトの利用状況とサブスクリプション状態の把握が優先されるためです。

```
サブスクリプション / 請求書 / 入金 → Billing
KPI / コホート分析          → DWH
顧客情報（補完的）           → CRM（軽量）
```

## 正本が複数存在するリスク

同じデータに対して複数のシステムがそれぞれ「自分が正本」と振る舞うと、一貫性が失われます。

```
よくある失敗例：

  顧客情報を CRM で更新した
    → Billing には古い情報が残ったまま
    → 請求書の送付先が間違い続ける

  Billing の商品名を直接変更した
    → CPQ の商品定義と名称がずれた
    → 受注報告書と請求書の商品名が一致しない
```

「正本は1つ」の原則を守るためには、データの更新はかならず正本で行い、他のシステムは同期によって最新状態を保つ設計を徹底します。同期の方向（どこからどこへ）と頻度（リアルタイム / バッチ）も、正本を決めた時点で合わせて設計する必要があります。

---

## このマトリックスの使い方

**データを更新したい**
→ 更新対象のドメインの「◎」のシステムで変更する。○のシステムでは直接変更しない

**システム間でデータがずれた**
→ 「◎」のシステムの値を正とし、○のシステムの同期状態を確認する

**新しいシステムを導入したい**
→ どのドメインの正本をそのシステムに移すのかを明確にし、旧正本からの移行計画を立てる

**データが「どこにある」かわからない**
→ まず◎のシステムを確認する。そこになければ、そのデータはまだ存在しないか、未定義のドメインに属している
